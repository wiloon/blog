## 背景

在业务增长阶段，很多系统都会遇到类似问题：写入路径以 `insert` 为主，单条记录字段很多，写入频率持续升高，最终出现写入延迟抬升、队列积压、应用侧内存压力变大。

这类问题通常被称为“宽表高频写入瓶颈”。这里把它拆开讲清楚：为什么会慢、先做什么优化、以及什么时候该考虑从关系型数据库迁移到时序数据库。

一个真实案例可参考：[物联网平台协议解析服务生产环境 OOM：MySQL 迁移 InfluxDB](../../career/iot-protocol-oom-mysql-influxdb.md)。

## 为什么宽表高频写入容易慢

宽表写入慢，通常不是单点问题，而是多个开销叠加：

1. 单行体积大：每次写入涉及更多页修改和日志写入。
2. 索引维护成本高：索引越多，单次 `insert` 需要更新的 B+Tree 越多。
3. 随机写放大：页分裂、二级索引回表结构维护，会放大 I/O。
4. 事务与刷盘约束：`redo/binlog/fsync` 的组合决定了吞吐上限。
5. 后台线程竞争：flush、checkpoint、purge 跟前台写入争抢资源。

这里的“更多页”并不是指“每一行一定跨很多页”，而是指一次写入会牵动更多数据页：InnoDB 以页（默认常见为 16 KB）为最小读写单位。行越宽，单页可容纳的记录数越少，同样数据量需要更多页；插入时更容易触发页满和页分裂。同时，写入不只改数据页，还会改聚簇索引页、二级索引页以及对应日志；如果包含较大 `TEXT/BLOB` 字段，还可能涉及溢出页。结果是单次写入的页修改与日志写入成本一起上升。

在高并发下，这些开销叠加后，常见表现是：写入 RT 上升，连接占用时间变长，线程池阻塞，进而把上游消费链路拖慢。

## 先做 MySQL 侧排查和优化

在决定换库前，建议先把 MySQL 侧可优化项跑一遍：

1. 表结构：拆分冷热字段，把大字段从核心写入表拆出去。
2. 索引策略：只保留必要索引，避免“为查而建”过多索引拖慢写入。
3. 主键设计：优先使用短且有序的主键，降低页分裂概率。
4. 批量写入：小批次 `batch insert`，减少事务提交次数。
5. 连接池与并发：避免连接池过大导致数据库抖动，结合压测设上限。
6. 存储参数：检查 `innodb_buffer_pool_size`、`innodb_flush_log_at_trx_commit`、`sync_binlog` 等配置是否与场景匹配。
7. 观测指标：重点盯写入 RT、TPS、磁盘写 IOPS、redo 生成速率、checkpoint age、锁等待。

如果这些动作做完后仍然无法满足目标吞吐，再考虑架构级改造更稳妥。

## 什么时候考虑时序数据库

当业务数据明显符合以下特征时，时序数据库通常更有优势：

1. 写多读少，且主要是追加写。
2. 读请求以时间范围查询为主。
3. 业务天然带时间戳主维度。
4. 接受按保留策略做冷热分层与过期清理。

这类场景下，时序引擎在写路径和压缩存储上的结构性优势，往往比继续堆关系库参数更有效。

## 迁移建议

从 MySQL 到时序库，建议用“双写 + 渐进切读”：

1. 先双写，确保新旧链路数据可对账。
2. 下游按服务分批切读，避免一次性切换。
3. 保留回退开关，出现异常可快速切回旧链路。
4. 观察一段稳定窗口后，再下线旧写入逻辑。
