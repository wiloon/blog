## 开链法与开放地址法，Separate Chaining and Open Addressing

哈希表在插入元素时，如果两个不同的 key 计算出的哈希值落到了数组的同一个位置，就发生了**哈希冲突（hash collision）**。解决哈希冲突主要有两种思路：**开链法（Separate Chaining）**和**开放地址法（Open Addressing）**。Java 的 HashMap 用的是开链法；Python 的 dict 用的是开放地址法。

## 开链法，Separate Chaining

开链法是一种二维结构：第一维是一个数组，数组的每个槽位（bucket）挂着一个链表（第二维）。插入元素时，先用 hash 函数算出该 key 应该落在数组的哪个槽位，然后直接把这个 key（连同 value）作为一个节点追加到该槽位的链表上。

当多个不同的 key 计算出相同的槽位时，它们并不会互相覆盖或者被挪到别处，而是全部挂在同一个槽位的链表里，按顺序排列。查找时，先定位到槽位，然后遍历该槽位的链表，逐个比较 key 是否相等，直到找到目标或者遍历到链表末尾。

### 优点

- 实现简单，冲突处理直观，槽位之间互不影响。
- 删除操作非常简单，只需要从链表中把对应节点摘除即可，不会影响其它元素的查找路径。
- 负载因子（元素个数 / 数组长度）可以大于 1，只是链表会变长，查找退化为线性扫描。

### 负载因子，Load Factor

负载因子的英文是 **load factor**，公式为 `load factor = 哈希表中总元素个数 / 数组长度（槽位数）`。这里的"元素个数"指的是**整个哈希表中的元素总数**，而不是某一个槽位对应的元素个数；某个槽位挂了多少个元素（链表长度）反映的是 hash 分布是否均匀，是另一个维度的问题。例如数组长度为 16，哈希表中一共存了 12 个元素，此时 load factor 就是 0.75。

开链法下即使 load factor 大于 1（元素个数超过数组长度）也不会出错，只是链表会变长；而开放地址法由于只有一个数组，load factor 不能超过 1（数组会满）。

### 缺点

- 每个节点都需要额外的指针字段存储下一个节点的地址，带来额外的内存开销。
- 如果 hash 函数分布不均匀，某些槽位的链表会很长，导致查找退化为 O(n)（有些实现在链表过长时会转成红黑树，比如 Java 8 之后的 HashMap，来缓解这个问题，详见 [HashMap](../language/java/hashmap.md#红黑树优化jdk-18-新增-treenode)）。
- 链表节点在内存中不连续，缓存局部性（cache locality）较差，性能上不如开放地址法这种连续数组结构。

Java 的 HashMap 就是使用开链法处理 hash 碰撞的典型例子。

## 开放地址法，Open Addressing

开放地址法与开链法的不同之处在于如何处理 hash 冲突。当新来一个元素哈希到数组中的位置已经被其它元素占据了该怎么办？

开放地址法只有一个数组（一维结构），会根据当前的位置计算出下一个位置，将这个冲突的元素挪进来。如果这下一个位置也被占用了，那么就再计算下一个位置，直到找到一个空的位置。可以想象，将会有一条虚拟的链条将这些相关的位置串起来。这个虚拟的链条就好比开链法里面的第二维链表，只不过链表有显式的指针字段，而虚拟链条没有，它完全是通过数学函数计算出来的。

```text
root = hash(key) % m               // 第一个位置，m 为数组的长度
index_i = (root + p(key, i)) % m   // 链条中的第 i 个位置

index_1 = (root + p(key, 1)) % m
index_2 = (root + p(key, 2)) % m
...
```

这个数学函数就是上面代码中的 `p` —— **probe sequence（探测序列）**。寻找空位置的过程就是一步一步的探测的过程，不同的 key 会生成不一样的探测序列。

在查找的时候，如果第一个位置上保存的 key 不是目标 key，那就沿着探测路径继续寻找，直到找到目标或者遇到一个空位置为止。

### 探测函数不能随意选择

探测过程有可能出现死循环——探来探去又回到原点，或者回到路径的中间。所以探测函数不能随意选择，它必须保证探测序列不会出现循环：经过 m-1 次探测生成的探测序列必须正好是 1..m-1 的全排列。

这样的探测函数有很多，其中最常见的一种是**线性探测函数**。该探测序列和输入 key 无关，最终的探测路径只和初始位置相关：

```text
// m = 2^n，c 必须是一个奇数
p(key, i) = c * i
index_i = root + c * i
```

这里不去仔细证明这个函数为什么满足要求，可以写个简单的代码来验证一下：

```java
public class HashTest {
    public static void main(String[] args) {
        int m = 1 << 16;
        int c = 111111;
        Set<Integer> nums = new HashSet<>();
        for (int i = 1; i < m; i++) {
            int p = c * i % m;
            if (nums.contains(p)) {
                System.out.println("duplicated");
                return;
            }
            nums.add(p);
        }
        System.out.println("no duplicate");
    }
}
```

输出结果：

```text
no duplicate
```

死循环的问题解决了。

### 删除的处理

开链法删除很简单，直接从链表中摘走就是；但开放地址法不能随意地将探测路径中的某个元素删除，这样会导致探测路径中断。为了不让探测路径中断，删除有两种实现方案：

1. **删除标记**：在删除的位置置一个特殊的删除标记，查找时可以直接跳过继续沿着探测路径往后寻找。这个删除的位置在后续新元素插入时会被回收利用：插入元素时遍历探测路径，遇到第一个删除标记的位置时不能立即插入，因为这个 key 有可能存在于探测路径的后半部；需要遍历到底确认路径里确实不存在这个元素后，再回过头插入到第一个删除标记的位置。如果删除的位置过多，会影响查找和插入性能。
2. **整体重建**：将探测路径的后半部元素全部删除，然后重新插入。如果路径较长，可能会影响插入性能。

### 聚集问题（Clustering）

在采用 `p(key, i) = c * i` 探测函数的前提下，如果多个不同的 key 哈希的第一个位置相同，那么它们将共享同一条探测路径——因为探测路径完全由第一个位置决定，和输入 key 无关。这些相关的 key 就会在一条探测路径上聚集，可能导致数据分布不均匀。

如果使用一个和输入 key 相关的探测函数，就可以消灭这个聚集问题：把探测函数中的常量 `c` 换成一个 hash 函数，只要这个函数总是返回奇数即可，这样的 hash 函数很容易编写：

```text
p(key, i) = h2(key) * i
```

Python 的 dict 就是使用开放地址法处理 hash 碰撞的典型例子。
