## 写作备注

- 项目发生在 2014 年，两次安全评估在同一年内完成（第一次约一周，几个月后第二次）
- APK 内部结构：正文中按实际流程描述（APK 是 zip → 解压得 classes.dex → dex2jar 转 jar → jd-gui 反编译），原始语音"APK实际上是jar包"是不精确的表达，请按技术准确的方式写
- 第一版 APK 发现了硬编码密钥、动态加载密钥等安全风险，但该 APK 并未对外公开发布，可以在文章中如实描述技术细节
- 第二版 APK 的加壳/脱壳防护方案，最终结论是"无 root 权限下安全"，客户通过行政手段禁止维修工程师的平板使用 root 账户作为最终解决方案，可如实写出
- 原始语音中细节描述存在口误和重复，生成正文时按合理逻辑整理，不必逐字还原

这是一个对安卓APK进行安全评估的项目。对方的公司是一个做电梯的公司。他们的电梯在安装之后要进行定期的维护和故障报修的处理。这个安卓APK是一个处理工单的系统。使用场景大概是电梯的维修工人拿着这个平板电脑，平板电脑上安装对应的安装这个APP，安装这个APK。然后系统会通过这个通过这个APK能够查看到电梯维修保养的订单工单。然后对方的公司委托我们公司对这个APK做一个安全评估，看看这个APK它是不是容易被破解，能不能，这个APK里面内置的一些机密的数据，比如说密钥，能不能被破解出来。这个项目时间不长，只有一周时间，属于是一个非常规的项目，因为我在第一家公司期间，一般的项目至少两三个月，项目周期长的可能要两三年。这个安全评估项目算是一个非标准的项目。部门领导把这个任务交给了我，可能是看我平时比较喜欢折腾各种技术，所以把这个破解的任务交给了我。这个项目本质上是对安卓APK进行安全评估，但是从另外一个角度讲的话，它是对这个APK做一个破解的测试，看看使用一些逆向手段、反编译的手段能不能够破解这个这个APK，拿到APK内置的一些机密的数据。我拿到这个APK之后，去网络上查了一些安卓APK逆向的一些，做逆向工程的一些知识。其实安卓APK在当年主要就是用Java语言编写嘛。然后它实际上是jar包。jar包呢就是zip包，zip包解压之后里面能看到class文件。然后再用一些Java的反编译的工具去反编译出源代码，然后再分析它里面的逻辑。我的blog里面，在同一时期应该记录了一些、收录了一些安卓APK做逆向工程的一些文档，可以关联到这个可以关联到正文里面。我拿到这个APK之后去查了一些资料，然后对APK包解压，对内层的jar包解压，拿到class文件，然后尝试了Java的各种反编译工具。反编译工具的文档，相关文档应该在我的blog里也有，可以找一下关联进来。用Java的反编译工具去反编译class，能看到分析出里面的逻辑。然后按照逻辑再继续分析的话，发现了硬编码在代码里的一些密钥，然后形成了第一版的我们叫可以叫安全评估报告。第一版的安全评估报告在一周之内提供给了客户。客户这个就是第一次的安全评估，为期一周。这个事情过去了几个月之后，这个客户又找到了我们部门，然后想让我们做第二次安全评估。我们拿到这个APK之后，发现它的保护方式有变化。因为第一版可能用的是比较简单的保护方式，只是一些第一版的APK应该是用了一些保护措施。他把一些逻辑封装成了架包，然后加包依赖在提供这个给给这个APK在用。只是很简单的保护措施，通过常规的反编译就能分析出来。好像也用了一些密钥的动态加载，就是说在程序运行起来之后，密钥才被释放出来才使用。但是呢，如果反编译拿到所有的源码，是能我当时是另外新建了一个Java工程，把里面的一段逻辑复制到这个Java工程里面执行。重复它的执行逻辑，最终拿到了被保护的密钥信息。就是说第一版的APK它是有一些保护措施的。然后几个月之后的客户又提供给我们的这一版新的版本的APK。应该能感觉到它的安全措施是升级过了的。猜测是猜测是第一版的评估报告，对方的公司，因为对方公司是做电梯业务的嘛，他的安卓APK应该也是找其他的第三方公司来做的。就是说这个电梯公司拿到评估报告，然后又去找了他们的vendor公司。我wonder公司根据我的评估报告，安全评估报告又重新升级了安全保护措施，才升级了第二版APK。然后这个电梯公司又把第二版APK拿给我们公司做安全评估。这个第二版的APK它用了类似病毒常用的计算机病毒常用的一种加壳脱壳的技术，最终的第二版APK安全评估结果是在安卓系统外层的安卓系统有。没有root权限的时候，用户没有外层安卓系统的root权限的时候，这个APK是安全的。之所以这么说，是因为安卓APK它的classloader类加载机制决定了这个问题，决定决定了这个问题。因为如果有root权限的话，安卓APK安装到安卓系统里面，它在运行的时候是是要把那个class文件还原到，还原出来到硬盘上，然后才能执行的，才能加载的。在加密状态下的class文件是不能加载的。然后第二版的APK使用了注册码的机制。就是说，呃呃电梯维修的人员在，或者IT人员在这个安卓平板上安装APK的时候，需要一个注册码。然后这个拿到注册码之后，这个程序这个安卓APK才能被使用。有了注册码，然后这个安卓APK的核心逻辑才能够脱壳出来，变成能执行的class文件或者jar包，才能够被加载，后续的逻辑才能被才能被执行。这就是两次的同一个安卓APK项目的两次安全评估，这个安卓APK安全评估的项目也是在我完全没有做过安卓APK开发的背景之下做的项目，但是好在因为我的是，我是Java背景的，我做过很多Java项目开发，然后又因为安卓底层是可以用Java，安卓底层的APK是安卓APK的底层开发是用Java语言的，所以但是安卓领域的话也有一些特别的领域知识。我在可以在短时间内去学习安卓APK逆向工程，然后完成这个项目。这个项目应该还是得到了用户的认可的，因为我的第二版的评估报告，因为第一版的评估报告的确是说明了一些问题，客户又让万道公司去升级APK。最终达到了一个可以接受的安全防护水平。后来客户针对第二版的安全评估报告给出的解决方案是，他们会有一些行政命令，因为他们的安卓平板是公司配发给维修工程师的，所以从IT基础设施上是能控制住的，就是说行政政策上禁止在这些平板上使用root用户，最终把这个问题解决掉了。这就是这个安卓APK逆向工程的两次安全评估，这个安卓APK逆向工程安全评估项目。

## 批注

- “几个月后，客户再次找到 vendor-H，带来了 vendor-A 升级后的新版 APK。” vendor-H 就是我所在的公司,所以这里的描述有点奇怪, 比如可以说客户再次找到我们公司.
- "采用了一种类似病毒防护常用的" 我理解加壳技术应该是病毒防止被查杀而使用的一种防护技术吧. 所以正文写的好像有点问题,你再看看
- 第二版 apk 使用了 注册码机制, 在没有注册码的情况下, apk 内部的逻辑是加密的, 只有 apk 在收到注册码 在运行的时候才会把核心逻辑释放出来然后收 class loader 加载和运行, 安卓的文件系统在非 root 权限的情况下是受保护的,所以理论上第二版是安全的.
- 另外写到这,我想到一个问题, 是不是可以再加一个防护,  apk 是否能检测运行环境,如果设备已经 root 了, 就拒绝运行, 不知道技术上能不能实现.
- “只有在运行时输入正确的注册码” 这里不准确, 应该是应用第一次安装的时候激活的时候
