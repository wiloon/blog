## A. 原始素材

> **AI 禁止修改本章节任何内容。**

### 2026-06-17

在楼兰时期的后期，大概是在2020年左右
在楼兰期间做过跟第三方外部系统的集成工作， 当时的背景是楼兰的车联网平台要跟第三方公司合作，楼兰提供车载设备和平台数据接口，第三方公司是做共享汽车业务的，第三方公司简称 M 公司，M公司采购楼兰的设备安装在共享汽车上，楼兰平台把这些设备对应的车辆数据通过接口提供给 M 公司，M 公司把这些数据提供给他们的客户（共享汽车用户），共享汽车用户通过 M 公司的 app 来查询车辆状态、行驶数据等信息。这个项目的集成工作主要是开发一个数据接口把楼兰平台上的数据，主要是前面提到的那个并发量最大的行车数据推送给 M 公司，主要的修改是在行车数据服务上按照车底底盘号（VIN）来区分不同的客户（M 公司和其他客户），把 M 公司的车辆数据发到楼兰平台内部的 kafka 上， 在kafka的另外一端用 go语言写了一个 consumer 来消费这些数据，做一些数据转换后通过 HTTP API 的方式推送给 M 公司,调用 m 公司的 api 把数据推送给 m 公司。
一些技术 细节和经验：
- 跟第三方系统的数据接口要打印和保留尽量全面的日志，每一批次的数据都要有唯一的可追踪的ID, 在早期的调试阶段和试运行阶段甚至要把日志级别调整到 debug级别，接口发送的 payload 要打印 全量日志，包括发送时间，trace id, payload, 对方 api的响应时间，状态码。
- 另外我还在这个 go 的消费端加了代码记录并发量和调用对方  api的响应时间等等监控数据， 这些监控数据 会被用于在楼兰平台上的 grafana 上展示监控数据。
- 以上这些都是为了在数据接口出现问题的时候方便快速的调查问题，定位问题，解决问题。
- 调试阶段遇到的问题，在项目第一次跟对方联调的时候出现一个特别神奇的问题， 并发量跑不上去，因为这个go写的 consumer在跟第三方联调之前我是内部测试 过的，并发量肯定是能满足要求的， 但是为什么 跟第三方联调就并发跑不上去呢， 肯定不是代码 问题，后来 才想起来，原来是我在内部测试 的时候 是公有云的内网，在准备跟第三方联调之前才给这个服务配置了公网出口， 又因为出口带宽会影响计费，所以在我们自己测试的时候只开了1M带宽的出口，结果在跟第三方联调的时候就发现并发量跑不上去，后来把出口带宽调高之后就好了，之所以没有第一时间想到这个问题，是因为内部测试的时候大并发量我们只在内网做了测试 ，开通外网出口之后我们只做了业务功能测试，没在做压力 测试。

### Round 1

**Q1:** 这个 Kafka consumer 是你独立开发的，还是团队协作？主要的技术决策（如选 Kafka 作为中间缓冲、用 Go 写 consumer）是你提出的，还是已有方案？

**作者回答:**
是我独立开发的， 并且 类似 的数据同步工具之前也开发过， 因为楼兰跟其它 第三方合作的话都 会有数据交互，之前也给其它 第三方开发过数据同步的服务，本次的 M公司是最大的一家
**Q2:** 行车数据服务按 VIN 分客户的改动，规模大不大？是在原有服务里加了一个过滤/分流逻辑，还是比较大的重构？

**作者回答:**
对于行车数据服务来说只是加了简单的分流 的逻辑，  VIN的映射关系是在 redis的， 服务收到数据之后 先查映射关系 再把特定的数据发送到特定的 kafka topic，另外一侧有对应 的第三业务 的 go 开发的 consumer + adaptor 来消费这个 topic的数据，做一些数据转换之后推送给 m 公司。
**Q3:** 联调时带宽问题排查出来之后，有没有做什么事后复盘或流程改进（比如以后外网测试必须做压测）？还是直接改了带宽就上线了？

**作者回答:**
改了带宽就继续联调了，后来也用这个配置用于生产运行，现在回看的话，当时在准备联调环境的时候准备出网通道的时候是考虑过带宽的问题的，因为担心高带宽费用高，当时也只是内部 测试 所以设置 了低带宽，当时 应该把这件事记录下来放在 backlog里，未来上生产的话带宽不够，这样就能在外部联调之前更容易看到这个问题，另外在设计阶段就应该把这个服务依赖的资源列一下，带宽可能要预先计算出来， 上线之前或者外部联调之前应该能有一个列表来检查这个服务依赖的资源是不是都 准备 好了。也许就能避免同样的问题再出现。

**AI Round 1 小结：**

素材已充分，整理后的技术全貌如下：

- **数据链路**：行车数据服务 → VIN 查 Redis 映射 → 写入 Kafka topic → Go consumer + adapter → HTTP POST 推送给 M 公司
- **作者职责**：独立开发 Kafka consumer + adapter，行车数据服务只做最小改动（加分流逻辑）；有同类项目经验，M 公司是规模最大的一次
- **可观测性**：trace ID 全链路日志 + Grafana 监控并发量和 API 响应时间
- **带宽排查故事**：内网测试通过 → 公网出口开 1Mbps（节省费用）→ 外部联调并发跑不上去 → 意识到带宽瓶颈 → 调高带宽解决；事后反思：应在设计阶段列出服务依赖资源清单，并记录潜在风险到 backlog
