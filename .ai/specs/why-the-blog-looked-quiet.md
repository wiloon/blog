## 1. 目的

向读者（以及未来回看的自己）说明：最近一段时间 wiloon.com 看起来像「停更」，并不是真的没更新，而是博客文章标题里的冒号触发 YAML 解析错误，导致 Cloudflare Pages 构建持续失败、域名一直停在上一次成功部署的旧版本。新文档其实已经写好并提交，只是没能发布出来。把这次「静默卡住」的经过讲清楚——范围限定在**最近这一段构建失败的时间**（约两天），不是「几个月」级别的长期停更。

## 2. 读者

- 主要写给未来回看这段时间的自己——这个 blog 平时读者很少，本篇定位更接近「给自己留一份故障记录」。
- 顺带交代偶尔回访、发现「最近 blog 没更新」的读者。

## 3. 核心信息（必须传达）

- 「停更」是错觉：新文章其实写了也提交了，只是因为构建失败没能上线。
- 直接原因：文章标题英中混写里出现「冒号 + 空格」（如 `Spring WebFlux: 响应式 Web 框架`），front matter 的 YAML 被解析成 mapping，`hugo` 构建中止，整站不生成。
- 失败是「静默」的：Cloudflare Pages 构建失败时保留上一次成功部署，wiloon.com 照常能访问、没有任何报错，只是内容停在旧版本——所以从读者视角看像停更。
- 一个意外收获（认知更新）：我原以为 Hugo 可以配置成「只渲染能解析的文档、跳过坏的那篇」，查下来发现**并不能**——任何一篇 front matter 解析失败都会中止整站构建，`ignoreFiles` 也救不了（Hugo 先解析所有 content 再应用忽略列表，见 [issue #10303](https://github.com/gohugoio/hugo/issues/10303)）。`ignoreErrors` 只覆盖少数特定错误 ID，不含 front matter 解析错误。
- 为什么 Hugo 这么设计（可简述，作者的理解）：Hugo 把整站当成一次编译产物，页面间有列表 / 分类 / 交叉引用等依赖，都要读每篇 front matter；若允许「跳过坏页、其它照渲」，结果会是列表悄悄缺页、分类计数错、引用断裂，而构建却显示成功——这种「静默残缺」比直接失败更糟。所以 Hugo 选择 fail-fast：一篇解析不了就当源码语法错误、中止整站，和编译器遇到语法错误就停同理。代价就是这次的「一篇写错、全站不更新」。

### 关联信息 · fail-fast

- 这次事故牵出的「Hugo 一篇坏、整站停」其实是 **fail-fast（快速失败）** 这个软件工程原则的一个体现。该概念已单独整理成文：[Fail-Fast 快速失败](../../content/post/pattern/fail-fast.md)（`content/post/pattern/fail-fast.md`），涵盖定义、常见例子、为什么这么设计、与 fail-safe/容错的边界等。
- **互链要求（输出/更新正文时务必落实）**：本篇正文**必须**在讲到 Hugo fail-fast 那段，用相对 `.md` 内链 `[fail-fast](../../content/post/pattern/fail-fast.md)`（成稿后相对路径为 `../pattern/fail-fast.md`）指向那篇——只链一次、放在最贴题处，**不展开概念**，概念解释交给 fail-fast 一文。
- 反向关联**已就绪**：fail-fast 一文已把「Hugo 构建中止」「构建失败要可见（邮件告警）」作为案例，回链到本篇与 [failover/failfast](../../content/post/inbox/failover.md)。故只差「本篇 → fail-fast」这一向，待正文补「认知更新」段时一并加上。
- 时间范围：这次卡住属于「最近一段时间」级别（约两天，2026-06-28 ~ 06-30 东八区，以 Cloudflare 控制台实际构建记录为准），**不是**「几个月」的长期停更。
- 当下的修复：给含冒号的 `title` 统一加引号，构建就恢复了。
- 针对这次故障，事后做了两件事（本篇的落点）：
  - **一是在 blog 的 AI 文档里加约束**：标题含「冒号 + 空格」必须加引号，并附上自查命令，把规则固化进 [AGENTS.md](../AGENTS.md) §标题规范 / §构建失败排查，防止再写出会让构建挂掉的标题。
  - **二是加邮件通知**：给 Cloudflare Pages 配了生产部署失败邮件告警（用 OpenTofu 管理，见私有仓库 w10n-config 的 `infra/cloudflare/opentofu/blog/main.tf` 里 `cloudflare_notification_policy.pages_build_failed`，按生产环境 `EVENT_DEPLOYMENT_FAILED` 触发）。以前失败站点照常、毫无提示；现在下次一失败就立刻收到邮件，不再静默卡在旧版本。
- （可选、简略）这段时间公开长文确实偏少、产出有所转移，但本篇主线是这次构建失败，转移方向只点到为止，细节交给 [exploration](../../content/post/career/exploration.md) 与周记。

### 方案选型 · 为什么最后选了邮件告警

> 解决「构建失败我一直不知道」时，当时一共摆了三个选项；最终没有选 AI 推荐的那个，而是选了邮件告警。正文要把三个选项和取舍都写出来，体现这是个权衡后的决定，而非随手一选。
>
> ⚠️ **待作者 review 确认**：当时 AI 推荐的具体是哪一个，需作者核对（从 fail-fast「越早暴露越好」的角度看，最可能推荐的是「提交前本地编译」那项；待确认后在正文写准）。

三个选项：

1. **Cloudflare 构建失败邮件告警（最终选中）**：不增加任何额外编译，构建一失败就收到邮件、第一时间知道。代价是它在「失败之后」才提醒——问题已经发生、已经 push，属于「快速知道」而非「提前拦截」，但对这个低频更新的个人博客足够了。
2. **让 GitHub（Actions）再编译一次做校验**：相当于在 Cloudflare 之外再跑一遍构建。虽然是免费的，但和 Cloudflare 的构建重复、感觉冗余，也白白浪费计算资源；更关键的是要**多维护一套 GitHub Action 定义**，平白增加维护成本——作者不想为此多养一条重复流水线。
3. **提交前本地编译一次**：最符合 fail-fast（在 push 前就把问题拦下、暴露最早）。但作者本地提交很频繁，本机性能又有限，编译一次要等很久；「频繁本地编译 + 长时间等待」正是作者不想要的，所以即便它能更早暴露问题也放弃了。

选邮件告警的理由：在「不增加重复构建、不拖慢本地提交节奏」的前提下，把原来最致命的「静默」这一点解决掉——让失败从「没人知道」变成「立刻有人知道」。它不是理论上最 fail-fast 的，但是是当前工作流下成本最低、最可持续的一个。
