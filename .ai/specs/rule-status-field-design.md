Rule 状态字段设计（第三家公司入职首月 2022年6月）

入职前几乎没有 Python 经验，入职约 1 个月后（2022-07）完成了第一个独立功能：为规则（Rule）增加状态字段（`status`），用于追踪每条规则在提交流程中的实时状态。这是后续所有状态机相关功能的基础。

**背景**  
入职前几乎没有 Python 经验，入职约 1 个月后，大家在讨论让我从哪个功能入手，这个在 UI 上显示 Rule 状态的需求是一个比较独立的功能，适合我这个阶段上手。

**面临的设计选择**  
给每条规则加状态，当时我们有两个选择：

当时有几种观点：
聚合方案修改的代码量最小，最容易改，如果用户之后有其它需求再说；
这个项目不确定还能做多久，不需要做那么完善；
状态是 Rule 的状态，逻辑上应该跟 Rule 在一起。设计理念是让问题回归它的本质，Rule 的状态就是 Rule 的状态，应该跟 Rule 在一起，而不是分散在其他表里通过 JOIN 聚合出来。

1. **SQL 聚合方案**（改动少，但有隐患）  
   不新增字段，通过 JOIN `rule_dryrun`、`rule_commit` 等多张表，在查询时实时计算出当前状态。改动代码文件少，但每次查询都需要跨表，逻辑分散，未来业务扩展时复杂度会持续积累。

2. **状态归属方案**（改动多，但清晰）  
   Rule 的状态就存在 Rule 表里（`rules.status`）。每当业务状态变更时，在对应的业务代码里更新状态字段。初期需要修改分散在多个模块的代码，但状态来源唯一，不依赖跨表聚合。

**选择理由和执行**  
我倾向于第二种方案，核心判断是：状态和数据应该同源，聚合查询的方案会在每一个新业务场景都"重新发明"一次状态计算逻辑，技术债随需求增长。  

**当时面临的真实阻力**  
这个决定并不容易做。当时我刚入职约一个月，Python 还没熟悉，和组内同事也还没有建立完整的信任关系。组内有多位同事倾向于保守方案（SQL 聚合）——部分原因也是考虑到我对代码库不熟悉，想减少我的上手压力，当时也不确定这个项目会做多久。

我表达了自己的判断之后，同事们表示了支持。一位熟悉代码库的同事专门指导我：rule 状态会在哪些业务节点发生变化、应该在哪些地方插入状态更新代码。功能最终成功上线，离不开同事和 leader 的支持。

这件事让我觉得值得记录的，不是"我说服了所有人"，而是：**在不熟悉语言、不熟悉代码库的情况下，我仍然清楚地表达了自己的技术判断，团队信任了这个判断并帮助我把它落地。**

**这是我在不熟悉 Python、不熟悉代码库、没有足够团队信任背书的情况下完成的，当时也没有 AI 辅助工具。**

**验证**  
这个设计经历了后续 3 年的持续迭代，所有后续的 Rule 状态机都是建立在这个字段基础上开发的，说明基础设计是正确的。
后来的 rule 状态统计报表的功能是在这个状态的基础上开发的。

**面试说法**  
> 我的第一个功能是给 Rule 加状态字段。当时有两种方案：SQL 聚合（改动少）vs 在 Rule 表加 status 字段（初期改动多）。同事们倾向保守方案，一方面是认为改动更小，另一方面也是考虑到我刚入职约一个月、对 Python 和代码库都还不熟悉，想减少我的压力。我表达了自己更倾向第二种方案，理由是聚合方案把复杂度延后了，每加一个新场景都会重新引入跨表依赖。同事们支持了这个判断——有一位还专门指导我 rule 状态会在哪些节点变化、从哪里插入代码。三年后回看，后续所有状态机相关的修改都建立在这个字段上修复，说明当初没有走弯路。

rule 状态字段上线的时候准备的 SQL 迁移（`migration.sql`）里同时为存量历史记录做了 backfill，backfill 正是通过复杂的跨表 JOIN 完成的——这也直接证明了如果不新增状态字段、只靠查询聚合，维护复杂度会有多高。

## 批注

- 把电商订单状态案例 和 银行账户余额案例， 在正文末尾加一个章节写一下，然后对比这两个经典案例
- rule 的状态字段上线之后我们又收到了几个需求，这些需求都只是 rule 的状态流转，让 rule 在不同的角色之间流转，但是这些操作都没有像 dryrun commit 一样记录操作历史，所以如果不是最初在 rule 上加状态 字段的话，到后期也会遇到问题再改成 rule 加状态字段， 后期改动量会更大。
- 这个故事里我想表达的另外一个点，在面对阻力的时候，坚持做对的事情，用第一性原理分析问题，清晰表达自己的判断获得团队的支持，最终把事情做好。不能因为短期便利或者对复杂性的顾虑而选择一个看起来更简单的方案，结果把复杂度留在后面，甚至让后续的需求都无法优雅地落地。面对不确定，面对复杂问题，要逐层分解，找到核心的本质问题，回归第一性原理，做出正确的设计决策。
- 另外在修改各个模块的时候不是在现有的业务代码里加 update 语句，我把 rule 状态做了封装，对于现有模块的修改一般只需要加一行函数调用。
