这个故事发生在我任职的第二家公司, 这有公司是做车联网的,我的角色是负责物联网平台的协议解析层的服务的开发和维护, 事件的起因是某一天收到生产环境的告警, 平台内部并发量最大的一个服务 内存告警, 这个服务部署了4个节点, 当时报OOM问题的时候 G4 服务总并发 5000 左右, 这个服务的业务是解析物联网设备上传的实时的车辆的数据,包括门,窗,锁,发动机的各种参数,车辆的各种液体的温度,油箱的剩余油量等等差不多两百个字段,这个服务负责解析这些数据然后把它们保存到平台内部的各种数据库里比如 redis , mysql 等等, 
我们收到这个服务的告警, 现象是4 个节点依次告警OOM, 节点1: OOM 然后,过一段时间恢复服务,然后节点2再 OOM,然后恢复服务, 节点3 OOM然后恢复服务,节点4 OOM然后恢复服务,然后再节点1又报OOM ,然后再到节点2 OOM,这样循环往复, 运维同事和我们开发的同事收到告警之后都登录生产环境去调查, 初步检查应用的日志和linux 系统的日志判断是应用内存占用过大被系统内核杀掉了,或者是被公有云监控到资源占用过大被公有云强制重启了虚拟机,重启之后这个节点会恢复服务一段时间, 但是内存占用会持续的增加直到下一次被杀掉或者被重启,又因为这个服务部署了4个节点, 一个节点挂掉之后其它三个节点的流量变大,导致了类似雪崩的效果, 三个节点的内存占用会快速的增长直到OOM.
当时没有太多时间深入调查根本原因, 但是初步判断这些流量都是正常的业务流量,不是 DDoS 攻击, 所以我们决定先扩容, 运维同事新建虚拟机,我去部署业务组件, 我们一边扩容一边观察监控数据,一直扩容到10个节点, 逐渐把上游消息队列里积压的数据消费完, 业务稳定之后,我开始调查OOM的问题,大概方向是调查这个服务在写入其它外部数据的代码函数执行时间,比如监控每批数据写入 redis, 写入 mysql  花费的时间, 大概方向是调查写入哪个服务有延迟导致数据积压在内存里消费不掉, 

当时有预期是 写 mysql 慢, 但是需要一些数据支撑

当时用了 Btrace这个工具,先在这个业务的外层函数加监控打印消费每一批数据的总时间,再向内层调查,调查写入每一种数据存储需要的时间比如写入 redis 花费的时间,写入 mysql 花费的时间, 当时大概判断是写入的mysql 延迟太大,导致数据积压,但是要有具体的数据支撑所以我用 btrace 在线织入的功能写btrace脚本逐个调查各个业务的写入延迟,结论是写入mysql占整个业务的比例最大, 后来 领导建议我们用 时序数据库来存这个实时采集的数据,我们决定逐步迁移到 Influxdb , 后来我去调查influxdb的官方手册在测试环境搭建 influxdb 做写入测试, 修改业务代码先实现双写,让业务数据同时写入 mysql 和 influxdb , 在过滤时期先双写业务数据, 联系消费这些数据的业务部门迁移到读 influxdb 直到所有依赖mysql 数据的业务都迁移到 influxdb ,我们平台再升级版本把写入mysql 的逻辑禁用掉,又过渡了一段时间, 直到完全删除写mysql的代码和回收mysql 数据库

脱敏与时间, 当时这个物联网,车联网平台使用的是 JDK8, 当时使用的 influxdb 版本大概是InfluxDB 1.5

公司名我觉得还是不出现好吧, 而且公司名云厂商跟这个故事的核心关联不大,云厂商是国内的一家当时刚进入公有云的一家厂商, 或者叫 R公司, 云厂商叫 "H云", 出问题的服务名字可以简写成 G4组件, 车联网平台可以称 R平台, 公司在大连, 可以写大连某车联网公司 或者叫 大连 R公司都行
1. 量化与现象, 700是 QPS, 700条数据每秒,峰值 1000+ 数据量是跟接入平台的同时在行驶状态的车的数量正相关的,因为车辆在行驶的时候会一直实时的上传数据到平台, 4到10节点扩容时间, 因为是运维同事批量操作,在云控制台批量创建虚拟机, 我们有自动运维系统, 然后在运维系统批量部署业务组件,这个扩容是小时级的,因为我们要保证尽量减少对线上业务的影响,简单判断都是正常的业务数据 不是攻击流量我们就快速扩容先扛住流量,保证业务响应时间,积压量级不记得了,但是发现的时候应该积压了一两个小时了再加上我们初步调查的时间大概估算是积压一两个小时的数据,按照峰值1000条每秒可以算一下,另外要补充的是协议解析层积压主要体现在 netty 的堆外内存数据积压,还有就是上游的接入层跟协议解析层的关系是推送数据的关系所以会表现成是协议解析层内存有积压数据消费不掉,当然了,协议解析层消费不慢应该也会影响接入层的推送,接入层的消息队列 rabbitmq里应该也有积压的数据, 节点OOM 的间隔是几分钟, 一个节点离线会导致其它节点的内存快速上涨,同时我们平台也会有来自公司客户服务部门的压力,因为平台出问题我们公司的400电话会被打爆,全公司都会关注我们的处理进度,所以我们选择了激进的快速扩容到10个节点,先保证业务响应为后面的调查争取时间,
节点重启之后也撑不了太长时间,也就是几分钟的级别因为上游也积压了大量的数据

## 批注

- “平台规模较大” 在正文里不要评判平台规模,只叙述事实，比如“平台共有300多台虚拟机”。
- “（峰值 1000 条/秒，可以算一下量级）” 这个信息是从哪来的, spec里没有, 也不应该出现在正文里
- OOM 之后, 在决定换 influxdb之前,也许 G4 的数据库连接池参数也是一个调查方向, 因为当时对于 mysql 的监控数据没有印象, 或许 mysql 的容量还够用, 但是相对于 关系型数据库来说, 时序数据库更适合我们的场景,所以我们没有继续在 mysql 的方向上继续优化, 选择直接换 influxdb 了, 这个可以在正文里补充一下当时的调查思路和为什么最后选择换库而不是继续优化 mysql 的参数配置之类的东西
