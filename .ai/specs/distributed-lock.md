多线程 Committer + 设备级并发锁（2023-02）
  
Committer 服务使用单线程顺序消费 Kafka 消息，单条提交若因设备响应慢而阻塞，整条队列都会等待。同时，没有防止同一台设备被并发提交的机制，存在竞态风险。
物理设备， 同一个设备 同一时刻只能有一个提交在进行中，否则可能导致 NSO 提交失败、设备状态不一致等问题。

- 将 Committer 改为多线程并发消费，Kafka 消费线程将消息提交给 `ThreadPoolExecutor`（默认 10 个 worker）异步执行
- 引入 `device_lock` 表作分布式设备锁：提交前先检查同一设备是否有正在进行中的提交，有则跳过或等待，完成后释放锁
- 锁的粒度为设备（device），不是请求（request），允许对不同设备并发提交

使用线程池处理策略并发提交
PostgreSQL NOWAIT 锁：设备锁使用 `SELECT FOR UPDATE NOWAIT`，竞争时立即抛出 OperationalError 而非阻塞，Kafka 消费线程可立即回到下一轮 poll 重试，不阻塞整个消费循环
Kafka 分区背压机制：高优先级消费线程获取设备锁失败时，会记录到内存中的 `blocked_kafka_partition` map；若所有 Kafka 分区都被阻塞，则把最旧的消息重新发送到 topic 尾部，释放分区，防止消费者卡死
finally 块保证锁释放：`unlock_device()` 放在 `finally` 块，无论 NSO 超时、解析异常还是任意未知错误，设备锁都一定释放，防止死锁

锁表方案简单可靠， 不引入 Redis 锁等外部依赖；`update_time` 字段记录锁变更时间戳，异常情况下可人工识别并清理过期锁



为什么用 PostgreSQL 表锁，不用 Redis 锁？
redis 的分布式锁的优势在于 TTL 特性， 可以自动释放过期的锁。 
Redis 主从同步默认是异步的，主节点写入后从节点尚未同步时若主节点宕机，锁状态会丢失，存在两个客户端同时持锁的风险。我们的 PostgreSQL 配置了 `synchronous_commit = on`，写入在主库和从库都落盘后才返回，锁状态具备强一致性保证。

为什么不用 PostgreSQL Advisory Lock？
Advisory Lock 是 session 级别的，session 结束自动释放，适合不需要跨请求持久化的短期锁。但设备锁需要在多个提交任务之间持久存在（一次 NSO 提交可能耗时数分钟），必须在数据库中有持久化记录，`device_lock` 表行锁的语义更合适。

已知问题：业务线程崩溃导致锁未释放
`unlock_device()` 在 `finally` 块执行，覆盖了绝大多数异常场景。但若进程级别崩溃（如 OOM Kill），`finally` 不会执行，锁会永久驻留。这种情况发生概率极低， 后续可考虑增加 APScheduler 定时任务，扫描 `update_time` 超过阈值的记录并自动解锁。
