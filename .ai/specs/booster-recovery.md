## 背景

**成文动机（作者）**：看火箭发射（Flight 12）时想到「早期不是已经夹通过几次吗，为何最近又总在落海」；当时**只猜测**过：一级这次不回收，会不会和二级要去的**测试轨道**有关。**从未**怀疑过「一级已经不行了」。与 **Gemini** 聊了一轮后整理成文——**不是官方口径，也不是定论**。

**作者真实疑问（交付物须对齐）**：早期数次塔捕（如 5、7、8）之后为何又多回落海？是否因二级要去测试轨道，一级才改落海？

**勿写入正文（仅 Spec 备忘）**：不要写「是不是一级不行了」类问句——作者未这样怀疑；也**不要**在正文里写「与一级能力不行无关」「我原本也没往这个方向想」等替自己辩解的句子，用 §误区二 等技术说明澄清即可。

**勿**把叙事收成「只有 Flight 5 夹成功过一次」。

交付物已按时间线与三类原因写清；本 spec **不重复正文**，只约束后续修订时的口径、语气与格式。

## 术语（一级落海 vs 着陆）

一级 **不回塔、在海上结束飞行** 时，中文优先用 **落海** 或 **海面溅落**，**不用「海面着陆」**——日常「着陆」易联想客机平稳接地或回发射塔捕获，与消耗式落海、hard splashdown 不符。

| 场景 | 推荐表述 | 避免 |
| ---- | -------- | ---- |
| 一级末段落墨西哥湾（统称） | 落海、海面溅落、不回塔 | 海面着陆 |
| 受控落水（soft / controlled splashdown） | 受控海面溅落、软溅落 | 软着陆（指一级时） |
| 复盘用语 | hard splashdown（可保留英文）或 硬溅落 | 着陆 |
| 程序名 | 着陆点火（Landing Burn） | — |
| Flight 4 路径 | 虚拟塔架着陆（业界常用说法，可保留）或 按塔架路径减速后溅落 | — |
| 正常回收 | 塔架捕获、回塔、RTLS | — |
| 任务表 Expended | 一级列为 Expended（消耗式、落海） | 一级着陆为 Expended |
| 计划内 vs 临场 | **计划内落海 / 海面溅落**；**临场分流落海** | 计划内海面着陆 |

二级 Ship 在印度洋溅落可用「溅落」；勿与一级「落海」混为一谈。

## 核心信息（修订时勿丢）

- 回收能力是**逐步解锁**的（空中 RUD → 海面虚拟着陆 → 塔捕），不是「测过塔架后放弃」。
- **塔捕与落海的时间线**：公开记录里 **Flight 5、7、8** 等曾成功「筷子」捕获一级；**Flight 5 之后多数试飞**一级为海面溅落或临场分流落海（如 6、10、11、12），并非「只成功过一次」。
- 一级 RTLS 与二级入轨**基本解耦**；热分离后一级 boostback，**不是因为**二级奔赴测试轨道而主动放弃一级回收（回应作者猜测，非「一级失效」）。
- 一级落海常见原因：**保护发射台**、**临场健康检查未过门槛**、**新箭型首飞计划内不 RTLS**（Flight 12 / V3 属此类）。
- **无「技术倒退」叙事**：星舰一直在迭代（V1→V2→V3、新 pad、新策略），从常理说**不适用**「能力退步 / 技术倒退」框架；落海是阶段选择与风险取舍，不是「夹不住了所以退步」。
- 当前测试重心：一级偏「求稳 + 迭代」，二级再入与复用仍是难点。
