# 文档长度与拆分

> **适用范围**：`content/post/**/*.md` 技术文章的篇幅控制与拆分决策。
> 与 [internal-links.md](internal-links.md)（拆分后内链）、[AGENTS.md](../AGENTS.md)（新建 / 编辑流程）配合使用。

## 行数参考（经验值，非硬性限制）

| 行数 | 状态 | 建议 |
| ---- | ---- | ---- |
| ~150–400 | 舒适 | 单一主题，正常维护 |
| ~400–800 | 偏长 | 留意是否已混入多个主题 |
| > 800 | 明显过长 | 大概率该拆分，主动提示作者 |

行数只是**触发信号**，不是判断依据本身。

## 真正的判断标准：单一主题 + 能独立阅读

是否拆分，看的是**一篇文章是否在讲多个相互独立的主题**，而不是单纯行数：

- 一篇 700 行但围绕单一主题、层层递进 → 可以不拆。
- 一篇 500 行却塞了 4 个互不依赖的主题 → 适合拆。
- 拆出来的每篇都应能**独立成文、被单独搜索到、有自己的 URL**。

## 拆分原则（hub 模式）

1. 每个独立主题拆成一篇，文件名 / 标题 / URL 语义化（见 AGENTS.md）。
2. **母文保留概览 + 链接**：母文留主题简介与跳转，正文细节迁到子文，形成「总-分」结构。
3. 子文之间、母子之间用**相对 `.md` 内链**互相串联，见 [internal-links.md](internal-links.md)。
4. 子文走**新建文章**流程：完整 front matter（`date`/`lastmod`/`url`）、`remix` + `AI-assisted` 标签。
5. 母文与被改动的子文都按需更新 `lastmod` 与 `## 维护记录`。

## AI 行为规则（重要）

遇到**特别长**（经验值 > 800 行，或明显涵盖多个独立主题）的文档时：

- **不要擅自拆分**。拆分会改变 URL 结构、内链与文章归属，属于「scope 变化」。
- **主动提示作者**：指出文档偏长、列出可独立成文的主题、给出拆分方案（哪些章节 → 哪个新文件 / 已有文件）。
- **由作者确认后再执行**拆分。
